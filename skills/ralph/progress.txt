## Codebase Patterns
- Test runner is `uv run --with pytest --with pyright pytest tests/` from the worktree root (no pyproject.toml; no venv on disk — uv resolves a one-off env each call).
- Typecheck is `uv run --with pytest --with pyright pyright`. pyrightconfig.json sets typeCheckingMode=basic, pythonVersion=3.10, includes core/tests/conftest.py.
- Coverage is `uv run --with pytest --with pytest-cov pytest --cov=core.<module>`.
- Tests MUST run via `wsl.exe -d Ubuntu -- bash -lc '...'` (Windows-native Python's uv can't import stdlib modules like `pwd` used by core/platform.py); the Windows path `\\wsl.localhost\ubuntu\...` only matters for file IO, not test execution.
- Frozen dataclasses for immutable value types (Config exception aside — it must be mutable for `replace()`). Use `@dataclass(frozen=True)` and tuple fields, not list.
- One module per concern under `core/`. Tests live at `tests/unit/test_<module>.py`; integration tests at `tests/integration/`.
- Internal helpers prefixed `_` and exported via `from core.<mod> import _helper` for direct unit testing — preferred over heavy monkeypatching when the helper takes parameters that drive its decisions.
- Canonical user-facing strings live as module-level constants (e.g. `WSL_LAUNCH_FIX_MESSAGE`, `NO_CLIPBOARD_WARNING`, `NoBackendError.MESSAGE`, `AUTH_ERROR_MESSAGE`) and tests assert on substrings of them to catch silent drift.
- File writes from PromptPal must use LF endings — `newline="\n"` on `open()` and `os.replace` for atomicity.
- conftest.py at repo root prepends the repo root to sys.path so `from core.<mod> import ...` works without installing the package.
- For modules with external I/O (HTTP, subprocess), expose injectable function kwargs in the constructor (`transport=`, `sleeper=`) — tests pass fakes that record calls. This keeps the production default sensible (`urllib.request.urlopen`, `time.sleep`) and avoids global monkeypatching.
- Bash scripts that ship to users (`bin/promptpal`, `install.sh`, the generated installed wrapper): use `set -euo pipefail`, pass `shellcheck`, keep LF line endings. Tests in `tests/unit/test_<script>.py` run the real script in a subprocess against a controlled HOME/PATH/INSTALL_DIR env and assert structural properties. Use `pytest.skip("shellcheck not installed in this env")` when `shutil.which("shellcheck")` is None so the suite stays green on hosts that don't have it.
- The WSL HOME guard is duplicated in every entry point (Python, bin/promptpal, install.sh, the installed wrapper). Every duplicate has a drift-fence test that asserts each line of `WSL_LAUNCH_FIX_MESSAGE.splitlines()` appears in its stderr. Edit one copy of the message → CI catches the others.
- When PATH-sensitive tests need to assert the "tool missing" branch, the test fixture's PATH must be `/usr/bin:/bin` (NOT `/usr/local/bin:/usr/bin:/bin`) — many dev/CI machines have a real `claude` in `/usr/local/bin/` that silently flips the assertion.

- Files changed: `core/api_backend.py` (new), `tests/unit/test_api_backend.py` (new), `ralph/prd.json` (US-004 → passes:true with notes), `skills/ralph/progress.txt`.
- **Learnings for future iterations:**
  - `ApiBackend.__init__` takes `transport=` (default `_default_transport` over urllib) and `sleeper=` (default `time.sleep`) — this lets every test run synchronously without network or wall-clock sleeps. The transport raises `_TransportNetworkError` for connection-level failures and returns `_HttpResponse(status=...)` for everything else (HTTP error codes included), so the retry loop has uniform input.
  - Retry counters per category are **independent** in this implementation (a 429 burst does not exhaust the 5xx budget). The PRD wording "max 3 retries" was ambiguous; tests (`test_retry_categories_have_independent_counters`) pin the chosen interpretation so a later refactor doesn't drift.
  - SSE parsing: the Anthropic stream uses `event: <type>\ndata: {json}\n\n`. The parser ignores `event:` lines and reads JSON from `data:` lines; the `[DONE]` sentinel terminates iteration. `_iter_sse` accepts any iterable of bytes lines, so tests use `io.BytesIO`.
  - Streaming usage: `message_start` carries `input_tokens` + initial `output_tokens=1`; `message_delta` carries the final `output_tokens` (and sometimes updated `input_tokens`). Both must be consumed to produce accurate token counts in the returned `BackendResponse`.
  - The `do_stream` decision is `bool(stream) and sys.stdout.isatty()` — when stdout is non-TTY (pipes, CI), `stream=True` silently falls back to a non-streaming request. Tests assert both branches by monkeypatching `sys.stdout.isatty`.
  - `urllib.error.HTTPError` IS the response object for non-2xx status codes — `.code`, `.headers`, and `.read()` all work on the exception itself. The default transport catches it and converts to `_HttpResponse(status=e.code, ...)` so the retry loop never sees an exception for HTTP errors.
  - The ANTHROPIC_API_KEY is read **only** in `__init__` (via `os.environ.get(..., "")`); empty/missing raises `ApiKeyMissingError`. The key is stored as `self._api_key` (private) and only appears in the `x-api-key` request header — never in `__repr__`, `__str__`, error messages, or retry-banner stderr lines. Test `test_key_never_leaks_to_stderr_on_auth_error` pins this.
  - `urllib.request.urlopen` should be monkeypatched as `api_mod.urllib.request.urlopen` so the test reverts cleanly. A fake response object needs `.status`, `.headers` (dict-like), and `.read()` for non-stream; for streaming it also needs to be iterable yielding bytes lines.

- Full project 549 tests green (was 516); pyright clean; coverage 96% across `core/` (install.sh is bash, not under coverage gate).
- **Learnings for future iterations:**
  - Install layout chosen: `~/.promptpal/{config.json, system-prompt.md, history/, lib/{core,defaults}/}`. The `lib/` subtree is *managed* — `rm -rf` + `cp -R` on every run so upgrades pick up new code. `config.json` and `system-prompt.md` are *user-owned* — guarded with `[ ! -e ]` to honor AC #4 "existing files never overwritten". US-015's `--purge` will target `lib/`, `history/`, and the launcher; the user-owned files get the prompt.
  - Generated installed wrapper is preferred over `cp bin/promptpal $INSTALL_DIR/`. Reason: `bin/promptpal` uses `cd "$(dirname "${BASH_SOURCE[0]}")/.."` to find the repo root, which breaks when the script is copied (the parent dir of $INSTALL_DIR has no `core/`) or symlinked (cd doesn't resolve symlinks). The generated wrapper hardcodes `PROMPTPAL_LIB="<abs path>"` at install time and exports it as `PYTHONPATH` before `exec python3 -m core.main "$@"`. Much more robust and lets `bin/promptpal` keep its dev-friendly relative-resolution logic unchanged.
  - Heredoc escaping for the generated wrapper: use **unquoted delimiter** (`<<WRAPPER_EOF`) so `$promptpal_lib` expands at install time; escape everything else with backslash — `\${HOME:-}`, `\${PYTHONPATH:+:\${PYTHONPATH}}`, `\$@`. A quoted delimiter (`<<'WRAPPER_EOF'`) would freeze the lib path as a literal `$promptpal_lib`. Verified by `test_generated_launcher_bakes_absolute_lib_path` which asserts `PROMPTPAL_LIB="<resolved tmp path>"` literally appears in the output.
  - `shellcheck install.sh` emits **SC2016** ("Expressions don't expand in single quotes") for two intentional cases: the `printf '  export PATH="%s:$PATH"\n'` copy-paste line and the `Option 1 (Claude CLI): ... `claude auth login`` line with literal backticks. Add `# shellcheck disable=SC2016` on the specific line above each — narrower than a file-level disable.
  - PATH-warning case match must use `:$PATH:` (with leading/trailing colons) to avoid false positives — `/home/.local/bin` would match `/home/.local/binary` without the colons. Pattern: `case ":${PATH:-}:" in *:"$INSTALL_DIR":*) ;; *) warn ;; esac`.
  - Test fixture `_base_path()` returns `/usr/bin:/bin` only — `/usr/local/bin` is omitted because some machines have `/usr/local/bin/claude` (system-wide install) that would silently make the "no backend → two-option hint" assertion fail by accidentally satisfying `command -v claude`. The has-claude tests prepend their own controlled shim dir to PATH; the no-backend tests don't get any extra entries.
  - Heredoc `$promptpal_lib` is a *lowercase* shell variable in install.sh because shellcheck/style discourages uppercase env-like names for non-exported locals. Inside the generated wrapper, `PROMPTPAL_LIB` IS uppercase because it's an env-style readonly. Don't conflate the two.
  - When asserting that an API-key-like string doesn't leak to stdout/stderr, set a deterministic fake value (e.g. `sk-ant-test-not-real`) in extra_env and grep for that exact string. Just asserting "no real key leaked" is a tautology.
  - `command -v <name>` is POSIX-portable and quieter than `which` (which may print errors to stderr on some distros). Always preferred in install scripts.
