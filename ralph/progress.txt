## Codebase Patterns
- Tests live in tests/unit/ and tests/integration/; root conftest.py puts the
  repo root on sys.path so `from core.x import y` works from any cwd.
- Atomic writes go through `core._io.atomic_write_bytes(target, payload, *,
  prefix, suffix)` (extracted in US-008). config.save_config,
  system_prompt._atomic_write_bytes, and history.write_session /
  upsert_index_entry all delegate to it. Tempfile prefix is per call site
  ('.config-', '.system-prompt-', '.session-', '.index-') so leftover-from-
  failure inspection can attribute by module. Tests that need to fault-
  inject the atomic write must monkeypatch `core._io.os.replace`, NOT the
  per-module os import (which doesn't exist anymore on these modules).
- Append-only NDJSON writes go through `core._io.append_ndjson_line(path,
  record)`. Not atomic by design — append semantics + one-record-per-line
  shape mean a kill -9 truncates at most the last record.
- All on-disk text is UTF-8 with LF endings (`newline="\n"` on text writes).
- preferred_backend is normalized through `normalize_preferred_backend()` at
  every entry point so a stale/corrupt value can never crash startup.
- Backend implementations inject their external dependency via a single
  `runner`/`transport` kwarg (ApiBackend → urllib transport; CliBackend →
  subprocess runner). Tests substitute a programmable Fake; production uses the
  module-level default. Keep this pattern when adding US-006+ resolve_backend.
- Subprocess-style backends translate FileNotFoundError → a typed `*NotFoundError`
  carrying the canonical SPEC §12 message; the CLI surface prints that message
  verbatim. Tests pin the message string so accidental drift breaks the suite.
- The Windows MSIX-launched session writes to WSL via UNC paths; pytest/pyright
  must run under WSL via uv. `.ralph-runner.sh` (gitignored) wraps the uv calls
  for each iteration: `bash .ralph-runner.sh test|typecheck|cov`. The
  typecheck command MUST include `--with pytest` so test files don't report
  spurious "Import pytest could not be resolved" errors.
- Backend resolution + persistence live in core/resolve.py. The CLI layer
  drives the persistence policy from outside resolve_backend: clear_backend_
  preference BEFORE resolution (when --backend auto), persist_backend_
  preference AFTER first successful turn (when --backend claude-cli|api-key).
  Failed turns must not call persist_* — the contract is structural, enforced
  by the caller's try/except, not by a guard inside the persist helper.

- History sessions live at `~/.promptpal/history/<uuid>.json`; the
  newest-first `index.json` is the search/list backing store; `usage.log`
  is append-only NDJSON. Session schema: session_id / created_at /
  updated_at / status / label / original_prompt / model / backend /
  turns[] / final_prompt. IndexEntry carries only session_id / created_at
  / label / status / original_prompt_preview (first 80 chars, no
  ellipsis — must match original_prompt[:80] byte-identical). Each Turn
  records role / content / backend / input_tokens / output_tokens /
  timestamp; tokens are None for CLI (P1-BKND-10), int for API.
- History writes are non-fatal (P1-HIST-08). The library raises OSError;
  the caller wraps with try/except and calls `warn_history_failure(err)`
  which prints the canonical P1-ERR-07 message to stderr. No sys.exit
  inside the history module.
- Test seam: every helper that touches `datetime.now()` / `uuid.uuid4()`
  accepts an injectable `clock` / `id_factory` so tests can pin
  timestamps and session ids without monkeypatching. Same shape as
  transport/runner/fetcher in earlier modules.
- Prompt input resolution lives in `core/input.py:read_prompt` and is
  pure-functional (returns a string, never sys.exit()s). Selection
  order: positional arg → stdin pipe (non-TTY) → interactive (TTY).
  Whitespace-only/empty input at any stage raises
  `EmptyPromptError(EMPTY_PROMPT_MESSAGE)` so callers can map a single
  failure mode to a single canonical stderr line + exit 1.
- Diff rendering lives in `core/diff.py` and is also pure: returns the
  diff string; the CLI layer prints it to stderr (P1-PIPE-09). The
  ">3 lines" threshold uses `len(improved.splitlines())` so a
  trailing-newline doesn't shift the count — `'a\\nb\\nc'` and
  `'a\\nb\\nc\\n'` are both 3 lines.
- Pipe-safety contract (P1-PIPE-09 — improved prompt to stdout, all
  else to stderr) is enforced by NOT writing to stdout from any
  helper module. read_prompt's interactive banner goes to stderr;
  format_diff returns a string. Keep this discipline in US-010
  (refinement loop) — choice line, spinner, raw print all go to stderr.
- The refinement loop lives in `core/loop.py:run_refinement_loop` and
  is pure-functional with full IO-seam injection (stdin / stderr /
  display / choice_reader / feedback_reader / copy_fn). It does NOT
  call sys.exit, does NOT write session JSON, does NOT call
  platform.copy_to_clipboard directly — persistence and exit codes are
  the CLI layer's responsibility (US-011); the clipboard goes through
  an injected `copy_fn` so tests don't need a real provider. Returns a
  LoopOutcome(status, final_prompt, messages, new_turns); the caller
  folds new_turns into the in-flight Session and writes it via
  history.write_session + finalize_session.
- The loop's `messages` is a list[dict[str, str]] projection of the
  conversation — caller seeds initial_messages with the original user
  turn + first assistant response; each [i]terate appends a user turn
  + assistant turn. Backend.complete is invoked with a DEFENSIVE COPY
  of messages so a misbehaving backend that mutates its argument
  cannot corrupt the loop's state (test pins this).
- Choice normalization is case-insensitive and FIRST-CHARACTER-ONLY:
  'accept yes' / 'A' / 'a' all map to ACTION_ACCEPT. EOF on stdin
  (readline → '') bails as STATUS_DISCARDED so a closed input stream
  cannot infinite-loop the test suite.
- The CLI layer (core/cli.py) is the SINGLE place that touches stdout
  for the improved prompt, the --output json envelope, and the
  --show-history / --search / --status / --export reports. Every other
  module writes diff chrome / banners / warnings to stderr. The pipe-
  safety contract (P1-PIPE-09 — improved prompt to stdout, all else
  to stderr) is enforced structurally by that one-place rule; a stray
  print() without file=stderr in core/* would break a pipe-safety
  test, not produce a silent bug.
- Backend.name has TWO canonical forms. The full form
  ("claude-cli (claude-sonnet-4-6)") is for human-readable display
  (--status). The short form ("claude-cli" / "api-key") is what the
  session schema, the index, the usage log, and the --output json
  envelope use (P1-BKND-10). core/cli._short_backend_name() strips
  the parens; do not store the full form in any persisted record.
- Test fixtures that seed ~/.promptpal/ inside tmp_path MUST override
  system_prompt_path in the config JSON to a tmp_path-relative path,
  otherwise Config's default value of "~/.promptpal/system-prompt.md"
  expands to the REAL $HOME at runtime and tests leak into the real
  config dir. Burnt one US-011 iteration on this.
- main() in core/cli.py accepts 14 injectable seam kwargs (config_path,
  history_dir, usage_log_path, env, stdin/stdout/stderr,
  detect_platform_fn, backend_factory, fetcher, clock, id_factory,
  copy_fn, skip_wsl_guard). When extending the CLI in US-012+ keep
  this seam set wide so integration tests stay filesystem-pure.

    refinement loop testable without a real history dir, and lets
    the CLI layer (US-011) decide things like '--no-history skips
    write_session entirely'. Don't fold session writes into the loop
    even though the AC text reads 'writes session JSON' — the
    structural contract is 'caller writes session based on outcome
    status', and the integration tests will pin that the CLI
    coordinates it correctly.

    will set copy_fn = functools.partial(copy_to_clipboard,
    platform=platform); platform.copy_to_clipboard prints its own
    NO_CLIPBOARD_WARNING on no-provider and returns False — DO NOT
    double-print the warning here. On the [c]opy menu action with
    copy_fn=None (clipboard hasn't been wired at all), the loop
    prints its own warning so the user knows the action was a no-op.
  - The choice line is the loop's structural contract — keep it in
    one place (CHOICE_PROMPT constant) and assert all 5 actions
    appear in it. If a future iteration adds [s]ave or [n]ame, the
    test that pins '[a]ccept [i]terate [d]iscard [r]aw [c]opy in
    CHOICE_PROMPT' will guide where the new action lands relative
    to the existing menu.

    they're being asked to accept' — one display call after the
    final auto-iteration, not one per round. This matches what a
    CLI user actually wants: they typed `--iterations 3` to skip
    the intermediate steps, so don't surface them.
  - Backend mutation defense (the defensive-copy in _iterate) is
    cheap and prevents an entire class of action-at-a-distance
    bugs. Adopt the same pattern in US-011 (CLI) when threading
    messages between layers — a list copy is O(n) of pointers,
    not contents, and the loop's messages list grows at most 2N+2
    deep for N iterations. The cost is invisible at any
    reasonable N.
  - Tests that use a queue-based choice_reader / feedback_reader
    self-terminate naturally: the queue empties → eof_reader
    returns '' → loop bails as discard. This is safer than test
    timeouts and means a misconfigured test (wrong number of
    inputs queued) fails loudly with a specific status assertion.
    Replicate this pattern when wiring tests for US-011's argparse
    layer.

- **Learnings for future iterations:**
  - The "pure-functional + injectable IO seam" shape that worked for
    backend transport / runner / fetcher works just as well for
    stdin/stderr/isatty. Tests construct io.StringIO instances
    locally rather than monkeypatching sys globals — no leakage
    risk between tests, parallel-safe.
  - "improved length > 3 lines" reads naturally as a > comparison
    against splitlines() count. Resist the temptation to count
    newlines directly: 'a\\nb\\nc' has 2 newlines but 3 lines, and
    'a\\nb\\nc\\n' has 3 newlines but still 3 lines. splitlines
    normalizes both shapes to 3.
  - The empty-prompt failure mode IS a separate error class
    (EmptyPromptError, not raised on a generic ValueError) so the
    US-011 CLI layer can catch it specifically and map to the
    canonical exit-1 + stderr message. Same pattern as
    SystemPromptMissingError / NoBackendError — each canonical
    user-visible error is one class with one constant message.
  - Whitespace-only positional arg raises rather than falling through
    to stdin. The reasoning: the user explicitly typed
    `promptpal ""` or `promptpal "   "`, which is almost always a
    shell-quoting bug; falling through silently would consume stdin
    they didn't intend to provide. This is the same defensive-default
    pattern as US-006's preferred_backend collapsing to 'auto' on
    unknown typo — explicit caller error → explicit explainable
    failure, never silent recovery into the next mode.
  - The pipe-safety contract (P1-PIPE-09) is structurally enforced by
    NOT writing to stdout in any helper module. US-010 (refinement
    loop) must continue this discipline — the choice line, spinner
    frames, raw print, and copy confirmations all go to stderr; only
    the final improved prompt lands on stdout (and only when the
    pipeline reaches the accept branch).

    dataclasses.replace-style mutators (append_turn returns a new

    canonical source of truth, the messages list is a projection.

    pick the no-op fallback over the fail-fast one.

    encode to bytes first, then delegate.

    preview to match original_prompt byte-identical at the front so

    normalization. If the future UI wants an ellipsis for display,
    it renders one at print time; the file-format value is unchanged.

    across system_prompt fetcher, history append_turn / new_session /
    finalize_session / append_usage_entry. US-009/US-010 should
    continue this pattern for any new module that depends on real
    time so tests stay deterministic.
